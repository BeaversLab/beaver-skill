# Beaver Markdown i18n

简体中文

## 功能描述

`beaver-markdown-i18n` 是一个专为 Markdown 文档设计的 token 高效翻译与同步流水线。它通过自动化脚本处理文档解析、占位符脱敏、翻译记忆（TM）匹配和质量校验，将复杂的 Markdown 结构转化为易于模型处理的“纯翻译任务”，确保在多语言本地化过程中代码块、变量、链接和 Frontmatter 结构绝对完整，并支持大规模文档库的增量同步。

## 优点

- **Token 高效性**：通过翻译记忆自动填充重复内容，并采用占位符脱敏技术，大幅减少发送给模型的不必要上下文。
- **结构零破坏**：利用 AST 解析和掩码技术（Masking），保护代码块、URL 和特殊变量在翻译过程中不被误改。
- **大文件友好**：支持自动分块（Chunking）处理，突破模型单次交互的上下文限制。
- **工业级质量控制**：内置 20+ 项自动化校验规则，强制拦截结构性错误和术语不一致。
- **增量同步能力**：支持基于 Git 提交或文件哈希的变更检测，仅针对更新部分触发翻译。

## 使用场景

### 适合使用

- **技术文档本地化**：特别是包含大量代码示例和技术术语的 API 参考、开发指南。
- **多语言文档同步**：源文档频繁更新，需要快速同步增量改动到多个语言版本。
- **大规模项目管理**：需要管理数百个 Markdown 文件的翻译状态和一致性。

### 不适合使用

- **全自动批量翻译**：本技能严禁不经人工/Agent 审查的全自动机翻，旨在辅助高质量人工/AI 协作。
- **非 Markdown 文件**：目前仅针对 Markdown 及其扩展语法（如 Frontmatter）进行了深度优化。

## 安装方法

本技能依赖 Node.js 环境及相关的 Markdown 解析脚本。安装技能后，请确保执行以下步骤以安装必要的依赖库：

```bash

# 安装依赖（推荐）

# 或者使用 npm

```

## 使用方法

### 触发方式

在对话中提及以下关键词或意图：

- “翻译文档”、“本地化”、“i18n”、“多语言同步”
- “翻译这个 Markdown 文件”、“同步中英文文档”

### 基本流程

1. **准备阶段**：执行 `prepare` 生成带有 `<!-- i18n:todo -->` 标记的骨架文件。
2. **手动翻译**：由 Agent 或用户针对标记区域进行翻译，严禁修改 `%%Pn%%` 占位符。
3. **完成阶段**：执行 `afterTranslate` 自动化完成还原、校验和状态更新。
   - _对于大文件，需在步骤 2 中逐个分块执行 `checkpoint` 并在完成后执行 `merge`。_

### 示例对话

> **用户**：请帮我把 `docs/guide.md` 翻译成中文，存放到 `docs/zh/guide.md`。
>
> **技能**：好的，我将启动翻译流水线。首先执行 `prepare` 操作。
> `node scripts/translate-cli.js prepare docs/guide.md docs/zh/guide.md --lang zh`
> [显示扫描结果：发现 5 个新段落，2 个已缓存。已生成骨架文件。]
> 现在我将开始逐段翻译标记的内容...

## 注意事项

- **占位符保护**：严禁删除或修改 `%%Pn%%` 和 `%%CB_hash%%` 占位符，否则文档将无法还原。
- **单分块原则**：在大文件处理时，每轮对话严禁读取或翻译超过一个分块，以防止上下文污染导致译文质量下降。
- **强制校验**：`afterTranslate` 报告 `ERROR` 时必须手动修复，报告 `WARN` 时需得到用户明确许可方可继续。

## 常见问题

**Q：为什么翻译后的文件里还有占位符？**
**A：** 因为你只完成了“翻译”步骤，还没有运行 `afterTranslate`（或 `apply`）。脚本需要读取占位符对应的原始数据来还原代码和链接。

**Q：如何导入已经存在的翻译版本到翻译记忆？**
**A：** 使用 `translate-cli.js seed <src_dir> <tgt_dir> --lang <lang>` 命令可以快速批量导入已有翻译对。
